**Target release:** May 14, 2026 (per plan-through-may-14).

## 1. Rate Limiting (W18 — Apr 27–May 3)

### Problem

A public dnsink deployment is an open DNS resolver. Two risks:

1. **Amplification DDoS.** Attacker spoofs source IP, sends small UDP queries, dnsink replies to the victim with bigger responses. Standard ~4× amplification, worse with DNSSEC.
2. **Cost blowout.** A sustained query flood on a hosted deployment (fly.io, anywhere) can blow through bandwidth quotas or auto-scale the bill. Observed real during v0.2.0 fly deploy cost analysis.

### Design

Per-source-IP token bucket. Each source IP gets a bucket of `N` tokens, refilled at `R` tokens/second. Every query costs 1 token. Buckets with 0 tokens get their queries dropped (UDP: silent drop; TCP: close connection).

**Why token bucket and not fixed window:** burst tolerance. A browser making 5 queries to resolve a page shouldn't trip the limiter; only sustained high-rate traffic should.

**Why per-source-IP and not global:** global limiter gets DDoS'd trivially. Per-IP scopes abuse cost to the attacker's botnet size.

### Config surface

```toml
[ratelimit]
enabled = true
requests_per_minute = 600   # 10 rps sustained
burst = 50                  # allow short bursts up to 50 queries

# Optional: exempt certain source IPs (internal networks, specific clients)
# exempt = ["127.0.0.1", "10.0.0.0/8", "192.168.1.5"]
```

Default: **enabled = false** (non-breaking for existing deployments). Users opt in.

### Module layout

- `src/ratelimit.rs` — new module. `TokenBucket` struct, `Limiter` holding `DashMap<IpAddr, TokenBucket>` (or `RwLock<HashMap>` if DashMap is new dep weight).
- Pluggable cleanup: evict buckets idle for >10 min to avoid unbounded memory growth.
- Wiring: `proxy.rs` UDP + TCP paths call `limiter.check(&src_ip)` before forwarding. Rejected queries increment a new metric counter and optionally log at trace level.

### Prometheus surface

New counter: `dnsink_ratelimited_total` — count of rejected queries. Dashboards can compare to `dnsink_queries_total` to measure abuse rate.

### Tests

- Happy path: well-behaved client stays under limit
- Burst: client can send `burst` queries rapidly, then slows to `requests_per_minute` pace
- Block: client exceeding rate gets dropped (returns None from limiter)
- Refill: token count regenerates over time
- Multiple clients: independent buckets
- Exempt list: 127.0.0.1 never rate-limited even under attack

### Open questions

- Do we want separate buckets per (IP, query_type) to prevent someone from burning their budget on one type and starving others? No — over-engineered. IP-level is enough.
- Should we return REFUSED instead of silent drop? Silent drop is the open-resolver norm; REFUSED leaks info. Start silent.
- `DashMap` vs `RwLock<HashMap>` vs sharded? Start with `RwLock<HashMap>`. Measure. Switch to DashMap if contention shows up in benchmarks.

## 2. DNS-over-TLS (DoT) upstream (W19 — May 4–10)

### Problem

dnsink supports UDP and DoH upstream. DoT (RFC 7858) is the third major encrypted DNS protocol and is the default on Android 9+ Private DNS. Users who want privacy-from-ISP but don't want the HTTPS overhead choose DoT.

### Design

Add `"dot"` as a valid `upstream.protocol` value alongside `"udp"` and `"doh"`. Uses `rustls` (already in the tree via reqwest's rustls-tls). Port 853.

### Config surface

```toml
[upstream]
protocol = "dot"                     # new value
address = "1.1.1.1"                  # resolver IP
port = 853                           # default for DoT
server_name = "one.one.one.one"      # required for TLS SNI + cert verification
# timeout_ms, existing
```

### Implementation

- New module `src/upstream_dot.rs` OR extend `proxy.rs` upstream handling with a trait for connection types.
- Connection pooling: DoT connections are expensive to establish (TLS handshake). Pool connections by (address, port, server_name) tuple, reuse.
- Idle timeout: close pooled connections after N seconds of no use.
- Wire format: DoT uses the standard DNS-over-TCP framing (length prefix). Existing TCP code paths should translate.

### Tests

- Happy path: resolve example.com via Cloudflare DoT (`1.1.1.1:853`, `cloudflare-dns.com`)
- Cert verification failure: connect to a bad SNI, fail cleanly
- Connection reuse: second query in same session uses cached connection
- Fallback behavior: if DoT connection fails, what happens? Options: fail the query, or fall back to UDP/DoH if also configured. For v0.3: fail the query (keeps behavior predictable).

### Open questions

- Do we support multiple upstream types simultaneously (e.g., DoT primary, UDP fallback)? Not in v0.3. Single upstream protocol per config. v0.4 could add an `[[upstream]]` array.
- Cert pinning? Not in v0.3. Rely on system trust store.

## 3. Config reload API (stretch — W20 — May 11–14)

### Problem

Currently, `dnsink` has hot-reload for blocklists (via `arc-swap`) but config itself requires a process restart. For a long-running daemon, changing feed toggles or tunneling-detection thresholds without downtime is valuable.

### Design

A signal-handler that re-reads `config.toml` and rebuilds configurable components. Initially triggered by `SIGHUP`.

```rust
// Simplified sketch
tokio::spawn(async move {
    let mut sighup = signal(SignalKind::hangup())?;
    while sighup.recv().await.is_some() {
        if let Ok(new_config) = Config::from_path(config_path) {
            config_swap.store(Arc::new(new_config));
            info!("config reloaded via SIGHUP");
        }
    }
});
```

### Scope limits

- **Reloadable:** feed toggles (enable/disable URLhaus mid-flight), refresh_secs, tunneling_detection threshold + min_subdomain_length, ratelimit requests_per_minute, metrics on/off
- **NOT reloadable:** listen.address + port (socket would need to rebind, messy), upstream.protocol changes (connection pool state), metrics.bind_addr

Validator rejects reload attempts that change non-reloadable fields: "Can't hot-reload listen.address; restart the process."

### Tests

Harder to test than features — involves signals and shared mutable state. Integration test that sends SIGHUP and verifies new config takes effect.

### Open questions

- HTTP control endpoint (`POST /admin/reload`) vs SIGHUP? HTTP is more flexible but adds auth burden. Start SIGHUP.
- What if new config is invalid? Keep old config, log error. Do not crash.

## Stretch — NOT in v0.3

Listed for reference; push to v0.4+:

- `hickory-server` integration for authoritative zone serving (different scope — dnsink is a proxy)
- Blocklist size persistence across restarts (currently re-fetches on boot)
- Prometheus histograms for latency (currently a counter-style cumulative; histograms are accurate but 10x more bytes on scrape)
- ECS (EDNS Client Subnet) support
- DNSSEC validation of upstream responses
